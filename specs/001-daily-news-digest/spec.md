# Feature Specification: AI 技术知识日报 Agent

**Feature Branch**: `001-daily-news-digest`
**Created**: 2026-05-15
**Status**: Draft
**Input**: User description: "基于 nanoclaw 的 cli-muyu group 中构建一个知识日报 Agent 新功能：每天早上9:00 在个人微信端收到一份昨日 AI 技术领域最值得关注的技术热点、新闻。核心功能包括 HN + RSS 抓取，LLM 摘要5条热点新闻，微信推送，设置定时任务"

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 每日自动接收 AI 技术日报 (Priority: P1)

作为关注 AI 技术动态的用户，我希望每天早上 9:00 自动收到一份昨日 AI 领域最值得关注的精选技术热点，无需手动操作，节省我每天浏览多个信息源的时间。

**Why this priority**: 这是整个功能的核心价值主张。自动定时推送解决了用户信息获取的痛点，是该功能存在的根本理由。

**Independent Test**: 可独立测试——手动触发一次日报生成流程，验证是否能成功抓取、摘要、分类并推送到微信，且内容格式正确。

**Acceptance Scenarios**:

1. **Given** 用户已完成日报 Agent 的初始化配置（包括定时任务设置和微信通道绑定），**When** 到达每天早上 9:00（Asia/Shanghai 时区），**Then** 用户在个人微信上收到一份结构化的 AI 技术日报，包含 5~6 条精选新闻，分 2~3 个主题板块，每条附原文链接。

2. **Given** 用户收到日报后，**When** 查看内容，**Then** 每条新闻都有清晰的标题、2~3 句话的核心要点摘要、以及可点击的原文链接。

3. **Given** 某天的数据源全部不可用，**When** 定时任务触发，**Then** 用户收到一条告知消息（如"今日数据获取异常"），而非静默失败或错误堆栈。

---

### User Story 2 - 灵活配置信息源 (Priority: P2)

作为用户，我希望能够自定义日报抓取的信息源，包括增减 RSS 订阅源，以适应我对不同技术领域的关注偏好。

**Why this priority**: 信息源的可配置性决定了日报的长期价值。不同用户关注的技术领域不同，固定源会限制功能的适用性。

**Independent Test**: 可独立测试——修改信息源配置文件后手动触发抓取，验证新配置的源是否被正确抓取和摘要。

**Acceptance Scenarios**:

1. **Given** 用户在配置文件中添加了新的 RSS 源地址，**When** 下次日报生成时，**Then** 新源的内容被纳入抓取和摘要范围。

2. **Given** 用户在配置文件中移除了某个 RSS 源，**When** 下次日报生成时，**Then** 该源的内容不再出现在日报中。

3. **Given** 某个配置的 RSS 源暂时不可用（如 5xx 错误或超时），**When** 日报生成时，**Then** 其他正常源的内容不受影响，日报仍能正常产出。

---

### User Story 3 - 查看任务状态和手动重试 (Priority: P3)

作为用户，我希望能够查看日报定时任务的运行状态，并在需要时手动触发一次日报生成，以便在特殊时刻（如重大技术发布日）立即获取最新汇总。

**Why this priority**: 状态可见性和手动控制能力提升了用户对系统的信任度，也是故障排查的必要手段。

**Independent Test**: 可独立测试——通过 Agent 交互查询任务列表，验证状态信息准确；手动触发后验证日报正常生成。

**Acceptance Scenarios**:

1. **Given** 用户向 Agent 询问日报任务状态，**When** Agent 响应，**Then** 用户看到当前定时任务的下次执行时间、最近执行结果等关键信息。

2. **Given** 用户要求 Agent 立即生成一份日报，**When** Agent 执行，**Then** 用户在数分钟内收到一份基于最新数据的日报，不影响原有定时任务 schedule。

---

### Edge Cases

- **数据源全部失效**：当 HN API 和所有 RSS 源同时不可用时，系统应优雅降级，向用户发送简短的错误提示，而非崩溃或无响应。
- **LLM 摘要失败**：当 LLM 服务不可用时，系统应发送原始标题列表（无摘要），保证信息可达性。
- **微信推送限流**：当微信通道触发频率限制时，系统应自动退避重试，并在多次失败后通过 CLI 通道通知用户。
- **内容重复**：同一条新闻可能在多个源中出现，系统应去重避免日报中出现重复内容。
- **时区变更**：如果用户更改时区配置，定时任务应按新时区的 9:00 执行。
- **LLM 输出格式偏离**：当 LLM 生成的日报格式不符合结构化要求时，系统应通过结构化输出约束（如 JSON schema 或 function calling）强制 LLM 返回标准格式数据，再由系统转换为 Markdown。若结构化约束仍失败，则发送原始标题列表作为兜底。
- **容器重启后任务恢复**：系统重启或容器重建后，已设置的定时任务应自动恢复，无需用户重新配置。
- **HN API 限流**：当 HN API 返回 429 时，抓取脚本应采用分批并发策略（每批 5 条，批次间间隔 200ms），避免触发 rate limit。若仍被限流，则指数退避重试最多 3 次。
- **手动触发与定时任务冲突**：当定时任务执行前 2 小时内已成功发送过日报（无论手动触发还是定时触发），本次定时任务应跳过执行，避免用户短时间内收到重复日报。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 支持定时触发日报生成任务，每天早上 9:00（Asia/Shanghai 时区）自动执行。
- **FR-002**: 系统 MUST 能够从 Hacker News API 抓取热门技术内容。
- **FR-003**: 系统 MUST 能够从配置的 RSS 源抓取内容，MVP 阶段至少支持 Anthropic Blog、OpenAI Blog、量子位、36氪 4 个源。
- **FR-004**: 系统 MUST 使用 LLM 对抓取内容进行智能筛选，从全部抓取结果中选出 5~6 条最值得关注的 AI 技术热点。筛选时优先选择昨日（前一天 0:00~24:00 CST）发布的内容；若昨日内容不足 5 条，则自动放宽至近 48 小时；仍不足时允许日报条目数相应减少。
- **FR-005**: 系统 MUST 对选出的热点进行主题自动分类，划分为 2~3 个板块（如 AI 大模型进展、产品与应用、研究前沿等），分类逻辑由 LLM 根据当日内容动态确定。
- **FR-006**: 系统 MUST 对每条热点生成 2~3 句话的核心要点摘要，保留原文标题，并附上原文链接。
- **FR-007**: 系统 MUST 将生成的结构化日报通过个人微信通道推送给用户。
- **FR-008**: 系统 MUST 支持通过配置文件自定义 RSS 源列表，包括添加、移除、修改源地址和抓取数量限制。
- **FR-009**: 系统 MUST 对多个数据源中重复出现的内容进行去重，避免日报中出现重复条目。去重判定标准为：normalized URL（去除协议前缀和尾部斜杠）完全一致视为重复；URL 不同但标题完全匹配也视为重复。
- **FR-010**: 系统 MUST 支持用户手动触发一次日报生成，独立于定时任务 schedule。
- **FR-011**: 系统 MUST 提供任务状态查询能力，让用户可以查看定时任务的执行历史和下次执行时间。
- **FR-012**: 当单条新闻的摘要生成失败时，系统 MUST 保留该新闻的标题和链接，仅跳过摘要部分。
- **FR-013**: 当全部数据源不可用时，系统 MUST 向用户发送预设的降级消息，告知数据获取异常。
- **FR-014**: 定时任务设置完成后 MUST 在系统重启或容器重建后自动恢复，无需重新配置。

### Key Entities

- **RawItem（原始条目）**：从各数据源抓取的单条内容，包含标题、链接、发布时间、来源、原始描述等属性。多个 RawItem 经筛选和摘要后成为日报条目。
- **DailyDigest（日报）**：一次日报生成的最终产物，包含生成时间、板块列表、每个板块下的条目列表（标题+摘要+链接）。日报通过消息通道推送给用户。
- **SourceConfig（源配置）**：定义一个数据抓取源的配置信息，包含源 ID、名称、类型（API/RSS）、URL、分类倾向提示、抓取数量限制。多个 SourceConfig 组成系统的信息源集合。
- **ScheduledTask（定时任务）**：定义日报生成任务的调度规则，包含任务名称、cron 表达式、时区、执行内容描述。任务由系统调度器管理，到期自动触发日报生成。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户每天早上 9:00（Asia/Shanghai）收到日报的成功率达到 95% 以上（以 30 天为观测周期）。
- **SC-002**: 单条日报的生成耗时（从定时触发到微信送达）不超过 5 分钟。
- **SC-003**: 日报内容覆盖至少 2 个不同的信息源（去重后），保证信息多样性。
- **SC-004**: 用户能够成功完成信息源配置的增删改操作，且变更在下次日报生成时生效。
- **SC-005**: 当单个数据源失效时，日报仍能正常生成并送达，成功率不因此下降。
- **SC-006**: 系统重启或容器重建后，已设置的定时任务在 2 分钟内自动恢复执行能力。

---

## Assumptions

- 用户拥有稳定的互联网连接，能够访问 Hacker News API 和配置的 RSS 源。
- 用户的微信已通过 iLink Bot 完成绑定和扫码认证，微信通道可用。
- MVP 阶段日报以纯文本/Markdown 格式推送，不包含图片、视频或其他富媒体内容。
- LLM 服务（Claude Provider）在正常负载下可用，API 调用有合理的成功率。
- 用户对 AI 技术领域有基本了解，能够理解日报中的技术术语和缩写。
- 信息源的 RSS 格式符合标准 RSS 2.0 或 Atom 规范，对于非标准格式采用最佳 effort 解析。
- 日报推送为单向信息推送，用户回复微信消息不会触发额外交互（MVP 阶段）。
- 数据持久化依赖 NanoClaw 内置的 SQLite 数据库（inbound.db / outbound.db）和容器文件系统（/workspace/agent/），无需外部数据库。

## Clarifications

### Session 2026-05-15

- **Q**: 日报内容的时效性过滤规则是什么？
  - **A**: 优先昨日（前一天 0:00~24:00 CST）内容，不足 5 条时自动放宽至近 48 小时，仍不足时允许日报条目数相应减少。
- **Q**: HN API 抓取策略与 rate limit 如何处理？
  - **A**: 采用分批并发策略（每批 5 条，批次间间隔 200ms），兼顾速度和稳定性。若触发 429，指数退避重试最多 3 次。
- **Q**: LLM 输出格式偏离时的处理策略？
  - **A**: 使用结构化输出约束（JSON schema / function calling）强制 LLM 返回标准格式数据，再由系统转换为 Markdown。若结构化约束仍失败，发送原始标题列表兜底。
- **Q**: 内容去重的判定标准是什么？
  - **A**: 以 normalized URL（去除协议前缀和尾部斜杠）去重为主，辅以标题完全匹配兜底。URL 或标题任一匹配即视为重复。
- **Q**: 手动触发与定时任务的时间冲突如何处理？
  - **A**: 同一自然日内，若定时任务执行前 2 小时内已成功发送过日报，则跳过本次定时任务，避免用户短时间内收到重复日报。
